# BUG-1 / BUG-2 / UX-3 Implementation Plan

> **For agentic workers:** REQUIRED: Use superpowers:executing-plans to implement this plan. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Fix two UAT-confirmed runtime bugs (stale detail view after save; blank Prayer Time cards) and add thin row dividers to the prayer card list.

**Architecture:** All three changes are isolated — two ViewModel fixes and one XAML tweak. No new files, no service-layer changes, no migrations needed.

**Tech Stack:** .NET MAUI 10, CommunityToolkit.Mvvm, NSubstitute (existing test framework)

---

## Root Cause Summary

### BUG-1 — Post-save view not refreshing

Navigation stack when editing a prayer from a card:
```
PrayerCardsPage → PrayerDetailPage [view-only] → PrayerDetailPage [edit]
```
After save, `GoToAsync("..?prayerSaved={id}&parentCardId={cardId}")` pops the edit page and delivers the query to its **immediate parent** — the view-only `PrayerDetailPage`.

`PrayerRequestDetailViewModel.ApplyQueryAttributes` only handles `"newForCard"` and `"load"`. It ignores `"prayerSaved"` and `"saved"`, so the view-only page never reloads → stale display.

`PrayerService.SavePrayerAsync` **does** correctly null both caches — the service layer is not at fault.

**Fix:** Add `prayerSaved` and `saved` branches in `ApplyQueryAttributes` that call `Reload()`.

---

### BUG-2 — Prayer Time blank card content

`PrayerTimeViewModel.LoadEntriesAsync` ends with:
```csharp
Entries = entries.AsReadOnly();
CurrentIndex = 0;           // ← BUG HERE
HasCompleted = entries.Count == 0;
```
`CurrentIndex` uses `SetProperty(ref _currentIndex, value)` from CommunityToolkit.Mvvm. Because `_currentIndex` starts at `0` (int default), `SetProperty(0, 0)` sees no change, returns `false`, and fires **no** `PropertyChanged` notification.

`CurrentEntry` is a computed property that depends on `Entries[CurrentIndex]`. Since it never receives a `PropertyChanged` notification after entries are loaded, every XAML binding (`CurrentEntry.CardTitle`, `CurrentEntry.PrayerTitle`, `CurrentEntry.Details`) remains at its initial null/empty state.

**Fix:** After setting `Entries`, bypass `SetProperty` by writing `_currentIndex = 0` directly, then manually raise `OnPropertyChanged` for all four dependent properties.

---

### UX-3 — Dividers between prayer request rows

`PrayerCardsPage.xaml` BindableLayout DataTemplate root is a `SwipeView`. Prayer rows have no visual separator between them.

**Fix:** Wrap each DataTemplate `SwipeView` in a `VerticalStackLayout` and append a `BoxView Style="{StaticResource DividerLine}"` below it.

## Chunk 1: BUG-1 — Stale detail view after save

**Files:**
- Modify: `PrayerApp/ViewModels/PrayerRequestDetailViewModel.cs`

### Task 1: Add `prayerSaved` / `saved` handlers to `ApplyQueryAttributes`

The existing `else if` chain in `ApplyQueryAttributes` handles `"newForCard"` and `"load"`. We add two new branches after the existing ones.

**Note on the ID guard:** `_prayer` is populated by `LoadPrayerAsync` which is fired as
fire-and-forget from the `"load"` branch. `ApplyQueryAttributes` is synchronous — if the
`prayerSaved` query arrives before `LoadPrayerAsync` completes, `_prayer.Id` will still be
`0` and a guard of `savedId == _prayer.Id` would silently fail. Since the view-only page
can only receive `prayerSaved` for the prayer it was already displaying, call `Reload()`
unconditionally.

- [ ] **Step 1.1: Locate the insertion point**

Open `PrayerApp/ViewModels/PrayerRequestDetailViewModel.cs`.
Find the closing brace of the `else if (query.ContainsKey("load"))` block (~line 272).
The two new `else if` blocks go immediately after.

- [ ] **Step 1.2: Insert the two new branches**

```csharp
else if (query.ContainsKey("prayerSaved"))
{
    // Navigated back to the view-only page after saving from the edit page.
    // Reload unconditionally — this page can only receive prayerSaved for
    // the prayer it was already displaying (no ID guard needed, and an ID
    // guard would be unreliable if LoadPrayerAsync hasn't completed yet).
    Reload();
}
else if (query.ContainsKey("saved"))
{
    // Same scenario via the PrayerListPage (ReturnToCards = false) code path.
    Reload();
}
```

Insert these after the closing brace of the `else if (query.ContainsKey("load"))` block and before the closing brace of `ApplyQueryAttributes`.

### Task 2: Fix `CurrentEntry` notification after `LoadEntriesAsync`

- [ ] **Step 2.1: Locate the assignment block**

Open `PrayerApp/ViewModels/PrayerTimeViewModel.cs`.
Find the end of the `try` block in `LoadEntriesAsync` (~lines 152–162):

```csharp
Entries = entries.AsReadOnly();

```

- [ ] **Step 2.2: Replace those three lines with**

```csharp
Entries = entries.AsReadOnly();
// SetProperty(ref _currentIndex, 0) is a no-op when _currentIndex is already 0
// (int default value). Bypass it and fire dependent notifications manually so
// the XAML bindings on CurrentEntry update correctly after the first load.
_currentIndex = 0;
OnPropertyChanged(nameof(CurrentEntry));
OnPropertyChanged(nameof(ProgressDisplay));
OnPropertyChanged(nameof(HasPrevious));
OnPropertyChanged(nameof(HasNext));
HasCompleted = entries.Count == 0;
```

### Task 3: Wrap each prayer DataTemplate item in a `VerticalStackLayout` + `DividerLine`

The BindableLayout DataTemplate currently has `SwipeView` as its root element.
Wrap it in a `VerticalStackLayout` and append a `BoxView DividerLine` after the SwipeView.

The line appears **below** each prayer row (including the last one, which precedes
the "+ Add prayer" row — visually fine since that row has its own top padding).

- [ ] **Step 3.1: Locate the BindableLayout DataTemplate in PrayerCardsPage.xaml**

Find (approximately line 88–131):
```xml
<BindableLayout.ItemTemplate>
    <DataTemplate x:DataType="viewModels:PrayerRequestDetailViewModel">
        <SwipeView Threshold="80">
            ...
        </SwipeView>
    </DataTemplate>
</BindableLayout.ItemTemplate>
```

- [ ] **Step 3.2: Wrap the SwipeView**

Replace the DataTemplate body:
```xml
<DataTemplate x:DataType="viewModels:PrayerRequestDetailViewModel">
    <VerticalStackLayout Spacing="0">
        <SwipeView Threshold="80">
            <SwipeView.LeftItems>
                <SwipeItems Mode="Reveal">
                    <SwipeItem
                        BackgroundColor="{StaticResource Primary}"
                        Command="{Binding EditPrayerCommand}"
                        Text="Edit" />
                </SwipeItems>
            </SwipeView.LeftItems>
            <SwipeView.RightItems>
                <SwipeItems Mode="Reveal">
                    <SwipeItem
                        BackgroundColor="{StaticResource DangerRed}"
                        Command="{Binding DeleteCommand}"
                        Text="Delete" />
                </SwipeItems>
            </SwipeView.RightItems>

            <!-- Prayer row -->
            <Grid
                Padding="16,8,12,8"
                ColumnDefinitions="*,Auto">
                <Grid.GestureRecognizers>
                    <TapGestureRecognizer Command="{Binding SelectPrayerCommand}" />
                </Grid.GestureRecognizers>
                <Label
                    Text="{Binding Title}"
                    FontSize="15"
                    TextColor="{Binding IsAnswered, Converter={StaticResource BoolToMutedColor}}"
                    TextDecorations="{Binding IsAnswered, Converter={StaticResource BoolToTextDecoration}}"
                    VerticalOptions="Center" />
                <Label
                    Grid.Column="1"
                    Text="{Binding AnsweredAtDisplay}"
                    Style="{StaticResource SuccessBadge}"
                    IsVisible="{Binding IsAnswered}"
                    VerticalOptions="Start"
                    Margin="6,0,0,0" />
            </Grid>
        </SwipeView>
        <BoxView Style="{StaticResource DividerLine}" />
    </VerticalStackLayout>
</DataTemplate>
```
