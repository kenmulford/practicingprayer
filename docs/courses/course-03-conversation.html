The Conversation

How components in the app talk to each other — data binding, commands, property notifications, navigation, and value converters.

In Course 2, you traced data through every layer. Now you'll understand how each layer communicates — the wiring that connects them. These patterns appear in every .NET MAUI app.

01

Data Binding — The Living Wire

A puppet and its strings — when the ViewModel moves, the UI follows.

The invisible string

Without data binding, you'd have to write code like "when the search text changes, manually update the search box." With binding, you declare the connection once and it stays alive forever.

One-way binding: ViewModel to View

CODE
<SearchBar
    Text="{Binding SearchText}"
    Placeholder="Search cards..." />
PLAIN ENGLISH

Create a search bar on screen.

Its text is bound to the SearchText property on the ViewModel. When the user types, the property updates. When the property changes, the search bar updates.

Show "Search cards..." as placeholder text when empty.

File: PrayerApp/Views/PrayerCard/PrayerCardsPage.xaml — lines 25–28

The ViewModel side of the binding

CODE
private string _searchText = string.Empty;
public string SearchText
{
    get => _searchText;
    set
    {
        if (SetProperty(
            ref _searchText, value))
            ApplyFilter();
    }
}
PLAIN ENGLISH

The actual search text is stored privately in _searchText.

The public property SearchText is what XAML binds to.

When the value changes, SetProperty stores it and notifies the UI.

Then immediately re-filter the card list to match what the user typed.

File: PrayerApp/ViewModels/PrayerCardsViewModel.cs — lines 72–77

💡
Binding is two-way by default for input controls

When you bind a SearchBar's Text, it's two-way: user types → property updates, property changes → search bar updates. For display-only elements (Label), binding is one-way by default.

02

Commands — From Tap to Action

Remote control buttons — each button triggers a specific function.

Why not call methods directly?

XAML can't call C# methods directly. Instead, the ViewModel exposes commands — objects that wrap methods. The XAML button binds to the command, and when tapped, the command fires the method.

The prayer detail page commands

CODE
public ICommand SaveCommand { get; }
public ICommand DeleteCommand { get; }
public ICommand MarkAnsweredCommand { get; }
public ICommand ShareCommand { get; }
PLAIN ENGLISH

SaveCommand — saves the prayer to the database.

DeleteCommand — deletes the prayer after confirmation.

MarkAnsweredCommand — marks the prayer as answered with today's date.

ShareCommand — generates a share link and opens the OS share sheet.

File: PrayerApp/ViewModels/PrayerRequestDetailViewModel.cs — lines 29–36

Two flavors of commands

RelayCommand

For instant actions that don't wait for anything. Example: toggling multi-select mode, dismissing a banner.

AsyncRelayCommand

For actions that take time (database, navigation). Automatically disables the button while running — preventing double-taps.

03

Property Change Notifications

A chain of dominoes — one change cascades to all dependent properties.

One change, many updates

When the overdue count changes on the home screen, three things need to update: the number, the headline text, and whether the "overdue" section is visible. OnPropertyChanged triggers all of them from one setter:

Cascading in HomeViewModel

CODE
public int OverdueCount
{
    get => _overdueCount;
    private set
    {
        if (SetProperty(
            ref _overdueCount, value))
        {
            OnPropertyChanged(
                nameof(HasOverdue));
            OnPropertyChanged(
                nameof(OverdueHeadline));
        }
    }
}
PLAIN ENGLISH

The overdue count property.

When a new value is set, store it and notify the UI.

If the value actually changed (not the same number again)...

Also notify the UI that HasOverdue might have changed (it's a computed bool: OverdueCount > 0).

And that OverdueHeadline might have changed (it's computed text: "1 Overdue" vs "3 Overdue").

File: PrayerApp/ViewModels/HomeViewModel.cs — lines 20–31

💡
Computed properties are free riders

HasOverdue and OverdueHeadline don't have setters — they're read-only formulas. They can't fire their own notifications. By calling OnPropertyChanged from the property they depend on, the parent "pushes" the domino for them.

04

Value Converters

A bilingual interpreter at the UN — translates between ViewModel language and UI language.

The translation problem

The ViewModel speaks in booleans: IsAnswered = true. But the UI needs a strikethrough, a gray color, or a flipped visibility. A value converter sits between them, translating one language to the other.

InverseBoolConverter

CODE
public class InverseBoolConverter : IValueConverter
{
    public object Convert(
        object? value, ...)
        => value is bool b && !b;
}
PLAIN ENGLISH

This converter flips a boolean. true becomes false, false becomes true.

Used to show elements when a condition is false: "show the empty message when there are NO items."

File: PrayerApp/Converters/InverseBoolConverter.cs — the complete file

BoolToTextDecorationConverter

CODE
public object Convert(
    object? value, ...)
    => value is true
        ? TextDecorations.Strikethrough
        : TextDecorations.None;
PLAIN ENGLISH

If the value is true (prayer is answered), show text with a strikethrough line.

If false (not yet answered), show normal text.

File: PrayerApp/Converters/BoolToTextDecorationConverter.cs

All six converters at a glance

InverseBool

true → false, false → true. Show things when a condition is not met.

StringToBool

"text" → true, "" → false. Enable a button only when a field has content.

BoolToTextDecoration

true → strikethrough, false → normal. Cross out answered prayers.

BoolToMutedColor

true → gray, false → normal color. Dim answered prayer text.

BoolToChevron

true → ▴ (up), false → ▾ (down). Expand/collapse indicator.

BoolToTriangle

true → ▶ (play), false → ▼ (down). Used in prayer time controls.

Navigating with data

Sometimes you need to tell the destination page what to show. Query parameters carry that data:

CODE
await _navigationService.GoToAsync(
    $"{Routes.PrayerTimePage}?scope={Routes.ScopeTags}&tagIds={systemTag.Id}");
PLAIN ENGLISH

Navigate to the Prayer Time page. Attach two pieces of info:

scope=tags — "show prayers filtered by tags, not all prayers"

tagIds=5 — "specifically, tag #5 (Recently Notified)"

IQueryAttributable

An interoffice memo system — the subject line tells you which action to take.

Receiving navigation data

The destination ViewModel implements IQueryAttributable to read the parameters:

CODE
void IQueryAttributable.ApplyQueryAttributes(
    IDictionary<string, object> query)
{
    if (query.ContainsKey("newForCard"))
    {
        if (int.TryParse(
            query["newForCard"].ToString(),
            out int cardId))
        {
            _prayer = CreateDefaultPrayer(
                cardId);
        }
    }
}
PLAIN ENGLISH

Shell calls this method after creating the ViewModel, passing the query parameters as a dictionary.

Check: did the navigation include a "newForCard" key?

If so, parse the card ID from the value.

Create a new default prayer pre-assigned to that card. This is how "Add Prayer" from a specific card works — the card ID travels as a query parameter.

Events & Callbacks

A newspaper subscription — subscribe to an event, get notified when it happens.

How the notification tap works

When a user taps a notification, the app needs to respond. This uses the event pattern — subscribe to a signal and react when it fires:

CODE
notificationCenter.NotificationTapped
    += async (_, notificationId) =>
{
    await App.InitTask;
    var systemTag = await
        tagService.GetSystemTagAsync(
            "Recently Notified");
    // ... show dialog, navigate ...
};
PLAIN ENGLISH

Subscribe to the "notification tapped" event. The += means "add me to the list of subscribers."

When a notification is tapped, this code runs.

Wait for the database to be ready (the app might have cold-launched from the notification).

Look up the "Recently Notified" system tag, then ask the user if they want to start a prayer time session.

File: PrayerApp/MauiProgram.cs — lines 214–235

Event-driven modals: the tag picker

The tag picker modal uses a more advanced event pattern. The ViewModel can't push a modal itself (that's a UI operation), so it raises an event delegate that the View subscribes to:

1
ViewModel declares the event

TagPickerRequested is a Func that the View will handle.

2
View subscribes in constructor

The code-behind wires up the event to push the modal page.

3
ViewModel invokes and awaits

The ViewModel fires the event and then awaits WaitForDismissAsync() — a TaskCompletionSource that resolves when the user taps Done.

4
Modal dismissed, caller resumes

The parent ViewModel syncs the selected tags back into its own collection.

💡
Why not just navigate with query params?

Query parameters work for passing IDs between pages. But the tag picker needs to return a list of selected tags back to the caller, and the caller needs to wait for the result. The event + TaskCompletionSource pattern solves both: the View handles the UI, and the ViewModel awaits the result without knowing anything about modals.
