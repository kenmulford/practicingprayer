06

The Bridge

How one codebase adapts to Android and iOS — conditional compilation, custom handlers, notifications, deep linking, and backup/restore.

Most of the app's code is shared between Android and iOS. But some things — notification channels, deep link intent filters, gesture handlers — require platform-specific code. This course shows how the bridge between shared and platform code works.

01

Conditional Compilation

A choose-your-own-adventure book — the compiler picks which pages to include based on the target platform.

The #if directive

Conditional compilation lets you write platform-specific code in shared files:

CODE
#if ANDROID
    PrayerApp.Platforms.Android
        .Handlers
        .TextInputTimePickerHandler
        .Configure();
#elif IOS
    PrayerApp.Platforms.iOS
        .Handlers
        .EnglishLocaleTimePickerHandler
        .Configure();
#endif
PLAIN ENGLISH

If building for Android: register the Android time picker handler (forces text-input mode for locale safety).

If building for iOS: register the iOS time picker handler (forces English locale for AM/PM display).

Each platform has its own quirk that needs a custom fix. The #if ensures only the relevant fix is included.

02

Custom Handlers

A tailor altering off-the-rack clothing — adjusting generic controls to fit each platform's quirks.

Why handlers exist

A handler connects a cross-platform MAUI control to its native implementation. The default handler works most of the time, but sometimes you need to fix a platform quirk:

📱
TextInputTimePickerHandler (Android)

Forces the time picker to use text input mode instead of the clock dial. Fixes locale-related formatting bugs.

🍎
EnglishLocaleTimePickerHandler (iOS)

Forces English locale on the time picker so AM/PM displays correctly regardless of device language.

📄
ModalPageSheetHandler (iOS)

Customizes how modal pages slide up — using sheet-style presentation for a more native feel.

The Switch thumb color fix

CODE
SwitchHandler.Mapper
    .AppendToMapping(
        "SyncInitialThumbColor",
        (handler, view) =>
{
    if (view is Switch sw && sw.IsToggled)
    {
        VisualStateManager
            .GoToState(sw, "On");
    }
});
PLAIN ENGLISH

Add extra behavior to every Switch control's mapping.

"SyncInitialThumbColor" is just a label for this customization.

If the switch starts already toggled on, manually trigger the "On" visual state. Without this, the thumb color doesn't match the style until the user physically toggles the switch.

03

Platform Services

Embassy translators — shared code speaks through platform-specific translators using a common language.

One interface, two implementations

Some features need native APIs. The pattern: define an interface in shared code, implement it separately for each platform:

CODE
#if ANDROID
    builder.Services.AddSingleton<
        IOrientationService,
        Platforms.Android.OrientationService>();
    builder.Services.AddSingleton<
        IColorPickerService,
        Platforms.Android.ColorPickerService>();
#elif IOS
    builder.Services.AddSingleton<
        IOrientationService,
        Platforms.iOS.OrientationService>();
    builder.Services.AddSingleton<
        IColorPickerService,
        Platforms.iOS.ColorPickerService>();
#endif
PLAIN ENGLISH

Android build: register the Android versions of orientation and color picker services.

iOS build: register the iOS versions instead.

The rest of the app just asks for IOrientationService — it doesn't know or care which platform it's on. The DI container handles the translation.

04

Local Notifications

A personal alarm clock collection — each prayer has its own clock with its own recurrence schedule.

How prayer reminders work

When a user enables notifications for a prayer, the app schedules local notifications based on the prayer's frequency:

Daily

One repeating notification. Fire at the set time every day.

Weekly

Repeats on a specific day of the week at the set time.

Monthly

12 individual one-shot notifications (one per month) — Android workaround for missing native monthly repeat.

One Time

Single notification that fires once and is done.

The scheduling code

CODE
public async Task ScheduleAsync(
    Prayer prayer)
{
    if (!_isNotificationsAllowed())
        return;

    _center.Cancel(prayer.Id);
    _center.CancelMonthly(prayer.Id);
    // ... schedule based on frequency
}
PLAIN ENGLISH

Schedule a notification for this prayer.

First check: are notifications even allowed? (The user can disable them in settings.)

Cancel any existing notifications for this prayer — both regular and monthly variants. We cancel before rescheduling because the frequency may have changed.

05

Deep Linking

A message in a bottle — a prayer encoded in base64, floated through the OS share sheet, decoded by the receiving device.

How sharing works

1
User taps Share

The app serializes the prayer to JSON and encodes it as base64.

2
URL is generated

https://practicingprayerapp.com/share/r?d={base64data}

3
OS share sheet opens

User picks a recipient — text message, email, etc. A fallback plain-text summary is included.

4
Recipient taps the link

If they have the app, it opens directly (via App Links / Universal Links). If not, a web fallback shows the content.

5
App decodes and imports

DeepLinkService parses the URL, decodes the base64 JSON, creates the prayer, and navigates to it.

Android intent filter

CODE
[IntentFilter(
    new[] { Intent.ActionView },
    Categories = new[] {
        Intent.CategoryDefault,
        Intent.CategoryBrowsable },
    DataScheme = "https",
    DataHost = "practicingprayerapp.com",
    DataPathPrefix = "/share",
    AutoVerify = true)]
PLAIN ENGLISH

Tell Android: "when someone taps a URL that starts with https://practicingprayerapp.com/share, open this app."

AutoVerify = true means Android checks the domain to confirm this app really owns those URLs.

06

Backup & Restore

A heart transplant in three stages — old DB out, new DB in, verify heartbeat.

Export: creating a backup

1
Close the database

Run a WAL checkpoint and close the connection so all data is in the main .db file.

2
Read the DB bytes

While the connection is closed (just milliseconds), read the entire database file into memory.

3
Reopen immediately

Reinitialize the database connection so the app keeps working.

4
ZIP and share

Package the DB bytes into a .pcrd file (which is a ZIP) and open the OS share sheet.

The export code

CODE
await _dbService.CloseAsync();

byte[] dbBytes = await
    File.ReadAllBytesAsync(_dbPath);

await _dbService
    .ReinitializeAsync(_dbPath);
PLAIN ENGLISH

Shut down the database connection cleanly (forces WAL data into the main file).

Read the entire database file into memory as raw bytes.

Reopen the database immediately — the app is only "offline" for milliseconds.

File: PrayerApp/Services/BackupService.cs — lines 42–48

⚠️
Why WAL checkpoint matters

SQLite's WAL mode writes changes to a separate file first. If you backup the .db file without checkpointing, you'll miss the most recent changes. The CloseAsync call forces all WAL data into the main file before we read it.

07

Entry Points

The front doors of two houses — Android's MainActivity and iOS's AppDelegate each welcome visitors in their own way.

Android: MainActivity

CODE
[Activity(
    Theme = "@style/Maui.SplashTheme",
    MainLauncher = true,
    LaunchMode = LaunchMode.SingleTop,
    ScreenOrientation =
        ScreenOrientation.Portrait)]
public class MainActivity
    : MauiAppCompatActivity
{ }
PLAIN ENGLISH

This is the Activity that Android launches first.

MainLauncher = true means this is the app icon's entry point.

SingleTop prevents duplicate activity stacks when a deep link arrives while the app is open.

Portrait locks the orientation — no landscape mode.
