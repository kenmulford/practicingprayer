The Safety Net

How automated tests, mocking, and continuous integration catch bugs before users do.

Tests are code that checks other code. Instead of manually tapping through the app after every change, automated tests verify hundreds of scenarios in seconds. This course teaches you how the Practicing Prayer test suite works.

01

Test Project Architecture

A flight simulator vs a real cockpit — linked files replicate the instruments without the actual airplane.

The linked-file pattern

The main app targets mobile platforms (Android + iOS). The test project targets plain net10.0 (no phone needed). But it needs to test the same code. The solution: linked files.

CODE
<!-- Models -->
<Compile Include="../PrayerApp/Models/Prayer.cs" />
<Compile Include="../PrayerApp/Models/PrayerCard.cs" />

<!-- Services -->
<Compile Include="../PrayerApp/Services/CardService.cs" />
<Compile Include="../PrayerApp/Services/IDBService.cs" />
PLAIN ENGLISH

Compile these model files from the main project as if they were in the test project.

Same for services. The test project gets the exact same code — no copies, no drift.

Files that depend on MAUI (Settings.cs, platform services, UI pages) are excluded — they can't run on a desktop.

02

xUnit Fundamentals

A quality control checklist at a factory — Arrange the product, Act on it, Assert the result.

Arrange, Act, Assert

Every test follows three steps, called AAA:

CODE
[Fact]
public async Task
    GetCardsAsync_ReturnsCardsFromDatabase()
{
    // Arrange
    var cards = new List<PrayerCard>
    {
        new() { Id = 1, Title = "Alpha" },
        new() { Id = 2, Title = "Beta" }
    };
    _db.GetAllAsync<PrayerCard>()
        .Returns(Task.FromResult(cards));

    // Act
    var result = await
        _service.GetCardsAsync();

    // Assert
    Assert.Equal(2, result.Count);
    Assert.Contains(result,
        c => c.Title == "Alpha");
}
PLAIN ENGLISH

[Fact] marks this method as a test that xUnit should run.

The test name describes what it checks: "GetCardsAsync returns cards from the database."

Arrange: Create fake data — two prayer cards. Tell the mock database to return these cards when asked.

Act: Call the real service method being tested.

Assert: Verify: we got 2 cards, and one of them is named "Alpha."

03

NSubstitute — Mock Everything

A stunt double in a movie — follows a script, and the director verifies the right stunts were performed.

Creating a mock

CODE
private readonly IDBService _db;

public CardServiceTests()
{
    _db = Substitute.For<IDBService>();
    PrayerCard.SetDBService(_db);
    _service = new CardService();
}
PLAIN ENGLISH

A private field to hold the fake database.

The test constructor runs before every single test.

Substitute.For creates a fake IDBService — a stunt double that looks real but follows your script.

Wire the mock into the PrayerCard model's static DB reference.

Create a real CardService. It will unknowingly use the fake database.

File: PrayerApp.Tests/Services/CardServiceTests.cs — lines 7–19

The two superpowers

.Returns()

Script the stunt double: "when asked for all cards, return this fake list." Controls what the mock gives back.

.Received()

Check the director's footage: "was GetAllAsync called exactly once?" Verifies that the code under test called the right methods.

Verifying the cache works

CODE
[Fact]
public async Task
    GetCardsAsync_SecondCall_UsesCacheNotDatabase()
{
    _db.GetAllAsync<PrayerCard>()
        .Returns(Task.FromResult(
            new List<PrayerCard>()));

    await _service.GetCardsAsync();
    await _service.GetCardsAsync();

    await _db.Received(1)
        .GetAllAsync<PrayerCard>();
}
PLAIN ENGLISH

Test: "calling GetCardsAsync twice should only hit the database once."

Tell the mock to return an empty list when asked for cards.

Call GetCardsAsync twice.

Assert: The database was queried exactly 1 time, not 2. Proves the cache worked — the second call used the cached result instead of hitting the database again.

File: PrayerApp.Tests/Services/CardServiceTests.cs — lines 52–60

04

Testing the Caching Pattern

Testing a vending machine — insert coin, press button, check the snack AND verify the warehouse call count.

What to test in a cached service

1
First call hits the database

Verify GetAllAsync was called (cache was empty).

2
Second call uses cache

Verify GetAllAsync was still only called once (cache served the second request).

3
Save invalidates the cache

After SaveCardAsync, the next GetCardsAsync should hit the database again.

4
Delete invalidates the cache

Same pattern: after delete, cache is null, next read goes to DB.

💡
This is what tests are for

Without these tests, a developer could accidentally remove the _cache = null; line and never notice. The app would look fine in manual testing (caches rebuild on restart). But the test catches it instantly: "Expected database to be called 2 times, but it was called 1 time."

05

Static State & Parallelization

A shared whiteboard in a conference room — serial meetings prevent mixed-up notes.

The problem

Remember the static _dbService reference on each model? In the constructor of every test class, we call PrayerCard.SetDBService(_db) to wire in the mock.

If two test classes run simultaneously, they'd each set a different mock — and both would be writing to the same static reference. Chaos.

The solution

Test parallelization is disabled for this project. Tests run sequentially — slower, but reliable:

⚠️
Gotcha: test passes alone, fails in CI

If you see a test that passes when you run it individually but fails in CI, the culprit is almost always shared static state. This is why parallelization is disabled here — and why a test you wrote might suddenly fail when someone else adds a new test class.

06

UI Testing with Appium

A robot tester with a touchscreen finger — a mechanical arm that taps buttons and reads screens.

How UI tests work

Unit tests check logic without a screen. Appium tests run the actual app on a real device (or emulator) and interact with it like a user:

1
Launch the app

Appium starts the app on a connected device or emulator.

2
Find elements

Locate buttons, labels, and inputs by their AutomationId.

3
Interact

Tap buttons, type text, scroll lists — exactly like a human user.

4
Assert

Verify the screen shows the expected text, elements, or state.

Unit tests vs UI tests

Unit Tests

Fast (milliseconds). Test logic in isolation. No phone needed. Run on every push. The first line of defense.

UI Tests

Slow (seconds per test). Test the full app on a real screen. Catches bugs that logic tests miss — like layout issues or navigation failures.

CI/CD with GitHub Actions

An automatic inspection station on a conveyor belt — every push triggers quality checks.

The pipeline

CI runs automatically whenever code is pushed:

CODE
on:
  push:
    branches: ["master", "dev", "feature/**"]
  pull_request:
    branches: ["master", "dev"]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - actions/checkout@v4
      - setup-dotnet@v4 ('10.0.x')
      - dotnet restore
      - dotnet build --configuration Release
      - dotnet test --collect:"XPlat Code Coverage"
PLAIN ENGLISH

Run this pipeline when code is pushed to master, dev, or any feature branch. Also on pull requests.

Job: run unit tests.

Use a Linux cloud server (tests don't need a phone — they run on plain .NET).

Checkout the code, install .NET 10, restore packages, build, and run all tests with code coverage.
