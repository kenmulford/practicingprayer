01

Variables & Types

The labeled jars in your pantry — each one holds a specific kind of ingredient.

Think of a pantry

Imagine a pantry with labeled jars. One jar says "Sugar" and can only hold sugar. Another says "Flour" — only flour. If you try to put sugar in the flour jar, it won't fit.

In code, a variable is a named jar. The type tells you what it can hold.

Variables in the Prayer model

Here are real properties from the Prayer model — the code that represents a single prayer request:

CODE
public int Id { get; set; }

public string Title { get; set; }

public bool IsAnswered { get; set; }

public DateTime? AnsweredAt { get; set; }
PLAIN ENGLISH

Id is a whole number. Every prayer gets a unique ID, like a ticket number at a bakery.

Title is text — the name the user gives this prayer, like "Health for Mom."

IsAnswered is a yes/no value — has this prayer been answered?

AnsweredAt is a date and time — when it was answered. The ? means "this might be empty" (unanswered prayers don't have a date yet).

File: PrayerApp/Models/Prayer.cs — lines 17–58

The four types you'll see most

string

Text — names, titles, messages. Always in quotes: "Quick Add"

int

Whole numbers — IDs, counts, positions. No decimal point: 42

bool

True or false — on/off switches. Is it answered? Is it a favorite?

DateTime

A specific moment in time. When was this created? When was it answered?

💡
Why this matters for AI

When you tell an AI "add a field to track how many times someone prayed," the AI needs to pick a type. Knowing that a count is an int (not a string) helps you verify the AI made the right choice.

Classes & Objects

Cookie cutters and cookies — the blueprint says what shape to make, each cookie is its own thing.

The cookie cutter

A class is a blueprint. It says: "every prayer card will have an Id, a Title, and an IsAnswered flag." But a class isn't data itself — it's the shape of data.

An object is one actual cookie. "Health for Mom" (Id: 7, IsAnswered: false) is one PrayerCard object. "Job Interview" (Id: 12, IsAnswered: true) is another — same shape, different data.

A real class declaration

CODE
namespace PrayerApp.Models
{
    [Table("PrayerCard")]
    public class PrayerCard
    {
        [PrimaryKey, AutoIncrement]
        public int Id { get; set; }

        [Column("Title"), MaxLength(100)]
        public string Title { get; set; }
    }
}
PLAIN ENGLISH

This code lives in the "PrayerApp.Models" folder of our project.

When saved to the database, store it in a table called "PrayerCard."

Here's the blueprint for a PrayerCard — anyone can use it.

Every card gets a unique number that counts up automatically.

Id: a whole number you can read or write.

The title column has a 100-character limit.

Title: text you can read or write.

PLAIN ENGLISH

Create a new PrayerCard and fill in its properties:

03

Collections

A playlist vs. a single song — sometimes you need to hold many items, in order.

One song vs. a playlist

So far, every variable holds one thing. But what about "all the prayer cards"? That's a collection — a container that holds many items.

The cache in CardService

The card service keeps a list of all prayer cards in memory so it doesn't query the database every time:

CODE
private IReadOnlyList<PrayerCard>? _cache;

public async Task<IReadOnlyList<PrayerCard>> GetCardsAsync()
{
    if (_cache is not null)
        return _cache;

    var list = await PrayerCard.LoadAllAsync();
    _cache = new ReadOnlyCollection<PrayerCard>(list);
    return _cache;
}
PLAIN ENGLISH

Keep a private read-only list of prayer cards. It might be empty (the ?).

This method returns all cards. It's async because the database call takes time.

If we already have the list cached, return it immediately — no database trip needed.

Otherwise, load all cards from the database and wait for the result.

Wrap the list so nobody can accidentally modify it, then remember it for next time.

Hand the list back to whoever asked.

File: PrayerApp/Services/CardService.cs — lines 14–26

Collection types you'll encounter

[ ]
List<T>

The everyday list — add items, remove items, access by position.

🔒
IReadOnlyList<T>

A list that can be read but not changed. Like a museum exhibit — look, don't touch.

#
Dictionary<K,V>

A lookup table — find a value by its key. Like a phone book: name to number.

Querying collections with LINQ

Once you have a collection, you often need to filter, count, or transform it. C# has a built-in tool called LINQ:

CODE
ActiveCardCount = activePrayers
    .Select(p => p.PrayerCardId)
    .Distinct()
    .Count();
PLAIN ENGLISH

Start with the list of active prayers...

From each prayer, grab just the card ID it belongs to.

Remove duplicates — we only want unique card IDs.

Count how many unique cards remain.

Enums & Constants

A menu with fixed options — you pick from a list, not free-text.

Why menus beat free-text

Imagine a form asking "How often should we remind you?" with a blank text box. Someone types "evry day," another "daily," another "each morning." Chaos.

An enum is a dropdown menu in code — a fixed set of named options.

This lives in the Models folder.

"One Time" (stored as number 0)

These are the only options. No misspellings possible.

05

Interfaces

A job description — defines what someone must do, not who does it.

The job posting

When a company posts a job, they list requirements: "must store data, retrieve data, and delete data." They don't care who fills the role.

An interface is that job posting. It lists what a class must do, but doesn't include actual code. The class that implements the interface fills in the details.

IDBService — the database contract

CODE
public interface IDBService
{
    Task<List<T>> GetAllAsync<T>() where T : new();
    Task<T> GetByIdAsync<T>(int Id) where T : new();
    Task<int> InsertAsync<T>(T item);
    Task<int> UpdateAsync<T>(T item);
    Task<int> DeleteAsync<T>(T item);
}
PLAIN ENGLISH

Here's the contract for any database service:

You must be able to get all items of any type.

You must be able to get one item by its ID.

You must be able to insert a new item.

You must be able to update an existing item.

You must be able to delete an item.

File: PrayerApp/Services/IDBService.cs — lines 16–24

💡
Why interfaces matter: swapping and testing

The real app uses DBService (talks to SQLite). Tests use a mock (a fake that pretends to be a database). Both implement IDBService, so the rest of the code doesn't know the difference.

Spotting interfaces in the wild

In C#, interfaces start with a capital I by convention:

I
IDBService

The database contract — implemented by DBService

I
ICardService

The prayer card contract — implemented by CardService

I
ITagService

The tag management contract — implemented by TagService

I
ISettings

The app settings contract — implemented by SettingsService

Async & Await

Ordering coffee and waiting for your name — you don't block the whole line.

The coffee shop

Imagine a coffee shop where you must stand at the counter, blocking everyone, until your latte is ready. The line would stretch around the block.

Instead, you order, get a number, and sit down. The barista calls your name when it's ready.

Async code works the same way. The app starts a database request, lets the screen stay responsive, and picks up the result when it's ready.

SaveAsync — saving without freezing

CODE
public async Task SaveAsync()
{
    if (Id == 0)
    {
        await _dbService.InsertAsync(this);
    }
    else
    {
        UpdatedAt = DateTime.Now;
        await _dbService.UpdateAsync(this);
    }
}
PLAIN ENGLISH

This method saves a prayer. async means "I might wait for something slow." Task is the receipt.

If the Id is 0, this prayer has never been saved.

await means "start the insert and wait, but let other code run."

Otherwise, this prayer already exists — we're updating it.

Stamp the current time, then await the update.

File: PrayerApp/Models/Prayer.cs — lines 75–88

The three keywords

1
async

Goes on the method declaration. Says "this method contains await calls."

2
await

Goes before any slow call. Says "start this, wait for it, but don't freeze the screen."

3
Task / Task<T>

The return type. Task = "I'll finish eventually." Task<List<Prayer>> = "I'll finish with a list of prayers."

⚠️
Common AI mistake: forgetting await

If AI-generated code calls an async method without await, the code fires and forgets — your data might not save. Always check that every Async method call has an await.

Generics

A vending machine that works with any coin type — one mechanism, many currencies.

One machine, many coins

Imagine building a separate vending machine for dollars, euros, and yen. Wasteful — the mechanism is the same.

Generics do the same in code. The <T> is a placeholder — fill it in when you use it.

GetAllAsync<T> — one method, every model

CODE
// The generic method in IDBService:
Task<List<T>> GetAllAsync<T>() where T : new();

// Using it for different types:
await _dbService.GetAllAsync<Prayer>();
await _dbService.GetAllAsync<PrayerCard>();
await _dbService.GetAllAsync<PrayerTag>();
PLAIN ENGLISH

This method can get all rows from any table. The T is a blank — fill in the type when you call it.

Get all prayers from the PrayerRequest table.

Get all prayer cards from the PrayerCard table.

Get all tags from the PrayerTag table.

Reading the angle brackets

Angle brackets < > always mean "of this type." Read them as "of":

T
List<Prayer>

"A list of prayers"

T
Task<int>

"A task that will produce an int when done"

T
IReadOnlyList<PrayerCard>

"A read-only list of prayer cards"

Static Members & Namespaces

A shared toolbox bolted to the wall vs. tools in your personal belt.

Wall tools vs. belt tools

On a construction site, some tools are bolted to the wall — everyone shares them. Others are in your personal tool belt — each worker has their own.

Static members are wall tools — shared by every object. Regular members are belt tools — each object gets its own copy.

The static database reference

CODE
private static IDBService? _dbService;

public static void SetDBService(IDBService dbService)
{
    _dbService = dbService;
}
PLAIN ENGLISH

A single, shared reference to the database service. Every Prayer object uses this same reference.

A one-time setup method. Called once at app startup, then every Prayer object can save and load itself.

Namespaces — organizing code into folders

A namespace is a folder label for your code:

📁

Data shapes: Prayer, PrayerCard, PrayerTag

📁
PrayerApp.Services

Business logic: CardService, TagService, DBService

📁
PrayerApp.ViewModels

Screen logic: HomeViewModel, PrayerCardsViewModel

💡
Namespaces = vocabulary for AI

When you tell an AI "put this in the Services namespace," it knows exactly where to create the file. Understanding namespaces helps you give AI precise directions.
