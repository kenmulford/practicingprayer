The Solution Map

A city zoning map — the solution is the city, each project is a district with its own building codes.

Three projects, one solution

The Practicing Prayer app isn't one giant pile of code — it's organized into three projects inside a solution:

📱
PrayerApp

The main app — all the code that runs on your phone (Android + iOS)

🧪
PrayerApp.Tests

Unit tests — automated checks that verify the logic without running on a phone

🤖
PrayerApp.UITests

UI tests — a robot that taps buttons and reads screens on a real device

Multi-targeting: one codebase, two platforms

The main project targets both Android and iOS from the same code. This is called multi-targeting. Most code is shared, but the Platforms/ folder has code that only runs on one platform.

💡
Why this matters

When you tell AI "add a feature," it needs to know: does this go in shared code (most things) or platform-specific code (like notification channels or deep link handlers)?

04

MauiProgram.cs

A spaceship pre-launch checklist — 15+ systems checks in exact sequence before liftoff.

The most important file in the project

MauiProgram.cs is the startup file. It runs once when the app launches and wires everything together. Think of it as the DI container setup — it tells the app "when someone asks for an ICardService, give them a CardService."

Step 1: Set the locale and init SQLite

CODE
var culture = new CultureInfo("en-US");
CultureInfo.DefaultThreadCurrentCulture = culture;

SQLitePCL.Batteries_V2.Init();

var dbPath = Path.Combine(
    FileSystem.AppDataDirectory,
    "prayer_app.db");
PLAIN ENGLISH

Force English formatting everywhere (dates say "Jan" not "Ene," times use AM/PM).

Turn on the SQLite database engine — must happen before any database code.

Build the path to the database file. It lives in the app's private data folder, named "prayer_app.db."

File: PrayerApp/MauiProgram.cs — lines 25–33

Step 2: Register services (the DI container)

CODE
builder.Services.AddSingleton<IDBService>(
    s => new DBService(dbPath));
builder.Services.AddSingleton<ICardService, CardService>();
builder.Services.AddSingleton<ITagService, TagService>();

builder.Services.AddTransient<HomeViewModel>();
builder.Services.AddTransient<MainPage>();
PLAIN ENGLISH

Register a single database service (singleton — one instance for the entire app).

When someone asks for ICardService, give them a CardService. One shared instance.

Same for tags — one TagService shared everywhere.

ViewModels are transient — a fresh copy every time a page opens.

Pages are also transient — rebuilt on each navigation.

File: PrayerApp/MauiProgram.cs — lines 125–201

Singleton

One instance, shared everywhere. Used for services and the database.

Transient

Fresh copy every time. Used for ViewModels and Pages.

Step 3: Wire the models to the database

CODE
var myDBService = app.Services
    .GetRequiredService<IDBService>();
PrayerCard.SetDBService(myDBService);
Prayer.SetDBService(myDBService);
PrayerTag.SetDBService(myDBService);
PrayerInteraction.SetDBService(myDBService);
CardBox.SetDBService(myDBService);
PLAIN ENGLISH

Grab the database service from the container.

Tell each model class "here's your database connection." Each call sets the shared static reference (from Course 0, Module 8).

After this, any PrayerCard, Prayer, or Tag can save and load itself.

File: PrayerApp/MauiProgram.cs — lines 238–244

05

App.xaml.cs & AppShell.xaml

A house with a hallway — App unlocks the door, AppShell is the hallway connecting five rooms.

App.xaml.cs — surprisingly small

CODE
public partial class App : Application
{
    public static Task InitTask { get; set; }
        = Task.CompletedTask;

    public App()
    {
        InitializeComponent();
    }

    protected override Window CreateWindow(...)
    {
        return new Window(new AppShell());
    }
}
PLAIN ENGLISH

This is the App class — the top-level entry point.

InitTask is a static "receipt" that pages await before loading data. MauiProgram sets this to SeedAsync so pages don't show blank content.

The constructor loads the XAML resources (colors, styles).

When the OS asks for a window, create one containing the AppShell — the navigation structure.

File: PrayerApp/App.xaml.cs — the complete file

AppShell.xaml — the five tabs

CODE
<TabBar>
    <ShellContent
        Title="Home"
        Icon="house_solid_full.png"
        ContentTemplate="{DataTemplate views:MainPage}"
        Route="MainPage" />

    <ShellContent
        Title="Prayer Cards"
        Route="CardsPage" />

    <ShellContent Title="Prayers" />
    <ShellContent Title="Tags" />
    <ShellContent Title="Settings" />
</TabBar>
PLAIN ENGLISH

Here are the tabs at the bottom of the screen:

Home tab — uses the house icon. ContentTemplate means the page is only created when the user taps the tab (lazy loading). Route name is "MainPage."

Prayer Cards tab — route name "CardsPage."

Prayers, Tags, and Settings tabs round out the five.

06

The Startup Timeline

A stage crew preparing for opening night — the set goes up first, props placed behind the curtain, actors wait for "places!"

What happens when you tap the icon

1
MauiProgram.CreateMauiApp()

Force English locale, init SQLite, build the DI container with all services.

2
Register global exception handlers

Set up crash logging so unhandled errors get written to the diagnostic log.

3
Wire models to database

Call SetDBService on each model class so they can save/load themselves.

4
App.InitTask = SeedAsync()

Start async seeding — create default colors, system tags, system boxes, Quick Add card. Runs in the background.

5
App constructor runs

Loads XAML resources (colors, styles).

6
AppShell renders the tab bar

The user sees the 5 tabs. The Home page is created lazily.

7
MainPage.OnAppearing awaits App.InitTask

The first page waits for seeding to finish before loading data. This prevents blank screens.

Why this order matters

💡
The key insight: show UI first, load data second

The app shows the tab bar before seeding finishes. The user sees a loading state instead of a blank screen. Each page awaits App.InitTask to ensure the database is ready before querying. This is a common pattern in mobile apps — show the shell immediately, fill in the content asynchronously.
