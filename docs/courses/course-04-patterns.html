01

MVVM — The Three-Layer Cake

A theater production — the script (Model) is the story, the director (ViewModel) decides how to tell it, the actors on stage (View) perform it.

What MVVM means

MVVM stands for Model-View-ViewModel. It's the organizing principle behind this entire app:

M
Model

The data shapes and database persistence. Prayer, PrayerCard, PrayerTag. Knows nothing about screens.

V
View

The XAML pages. Buttons, labels, lists. Knows nothing about business logic — only displays what the ViewModel provides.

VM
ViewModel

The brain in between. Receives user actions (via commands), talks to services, exposes data for the View to display (via properties).

💡
Why this matters for AI

When you tell AI "add a delete button," it needs to know: the button goes in the View, the delete logic goes in the ViewModel, and the data deletion goes in the Model/Service. Saying "follow MVVM" instantly communicates this three-way split.

02

Active Record

Self-filing documents — each model knows its own filing cabinet drawer and can walk itself there.

The pattern

In the Active Record pattern, each model object carries its own persistence logic. You don't need a separate "PrayerCardRepository" — the PrayerCard itself knows how to save and load.

CODE
// The model saves itself:
await card.SaveAsync();

// The model loads itself:
var card = await PrayerCard.LoadAsync(42);

// The model deletes itself:
await card.DeleteAsync();
PLAIN ENGLISH

Save this card to the database (it knows how).

Load prayer card #42 from the database.

Delete this card from the database.

Each of these is an instance method on the model itself. The model is the document that files itself.

The trade-off

Pros

Simple, intuitive, less boilerplate. One class holds everything about a concept.

Cons

Models depend on the database (via static _dbService). Harder to test in complete isolation. Less flexible for complex queries.

03

Singleton Caching

A librarian with a clipboard — first request reads the shelves, subsequent requests use the clipboard.

The read path

1
Someone asks for all cards

GetCardsAsync() is called. Is the clipboard (_cache) filled?

2a
Cache hit: Return the clipboard

The list was already loaded — return it instantly. No database trip.

2b
Cache miss: Read the shelves

Load all cards from SQLite, copy them onto the clipboard, return.

The write path

1
Someone saves or deletes a card

The database changes.

2
Throw away the clipboard

_cache = null; — the old list is stale.

3
Next read rebuilds

The next GetCardsAsync() call sees null, goes to the database, and rebuilds.

⚠️
The #1 caching bug: forgetting to invalidate

If you add a save method but forget _cache = null;, the new data is in the database but invisible to the UI. The screen shows stale data until the app restarts. Always check: "does this write method clear the cache?"

04

Interface Segregation

A job description vs the employee — swap the person, keep the contract.

Every service has a contract

The interface segregation pattern means the rest of the code never knows (or cares) which specific class is doing the work:

I
ICardService (contract)

Defines: GetCardsAsync, SaveCardAsync, DeleteCardAsync, InvalidateCache

C
CardService (real implementation)

Talks to SQLite. Used in the running app.

M
Mock CardService (test fake)

Returns pre-set answers. Used in unit tests. Same contract, different behavior.

Cascading Operations & System Guards

Removing a load-bearing wall — cascade deletes clear dependent records; guards protect critical structures.

Cascade deletes

When you delete a tag, what happens to all the prayers tagged with it? A cascade delete cleans up the junction records:

1
User deletes a tag

TagService.DeleteTagAsync(tagId) is called.

2
Clear all junction rows first

DeleteByTagIdAsync removes all PrayerCardTag rows pointing to this tag.

3
Then delete the tag itself

The PrayerTag row is removed. No orphans left behind.

System entity guards

Some records must never be deleted. System guards protect them:

🛡️
Protected entities

The "Quick Add" card, "Recently Notified" tag, "System" box, and "Archived" box are all marked IsSystem = true. Delete methods check this flag and refuse to proceed. This prevents accidental destruction of built-in features.

06

Computed Properties

A spreadsheet formula — change one cell, all dependent cells update.

Properties that derive from other properties

CODE
public bool HasOverdue
    => OverdueCount > 0;

public string OverdueHeadline
    => OverdueCount switch
{
    0 => "All requests have been recently prayed for.",
    1 => "1 Overdue",
    _ => $"{OverdueCount} Overdue"
};
PLAIN ENGLISH

HasOverdue is true if the count is greater than zero. It has no setter — it's a formula.

OverdueHeadline generates different text based on the count. 0 = encouraging message. 1 = singular. 2+ = plural with the actual number.

The switch expression (switch {}) maps the count to the right text.

07

Error Handling & Diagnostics

A black box flight recorder — records the last events before every crash.

DiagnosticLog — the crash recorder

CODE
public void Log(
    string category, Exception ex)
{
    var entry = $"[{DateTime.UtcNow:yyyy-MM-dd HH:mm:ss}] [{category}] {ex.GetType().Name}: {ex.Message}\n{ex.StackTrace}";
    lock (_lock)
    {
        File.AppendAllText(
            _logPath, entry);
    }
}
PLAIN ENGLISH

Write a crash entry with the timestamp, category (like "UnhandledException"), error type, message, and the stack trace.

The lock prevents two threads from writing to the same file simultaneously (which would corrupt the file).

Trim — preventing unbounded growth

CODE
public void Trim()
{
    if (entries.Length <= MaxEntries)
        return;

    var trimmed = entries
        .Skip(entries.Length - MaxEntries);
    File.WriteAllText(_logPath,
        string.Join(Delimiter, trimmed));
}
PLAIN ENGLISH

If there are fewer than 100 entries, do nothing — the log is small enough.

Otherwise, keep only the most recent 100 entries (skip the oldest) and rewrite the file.

This runs once at startup (in SeedAsync) to prevent the log file from growing forever.
