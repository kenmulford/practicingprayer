03

The Service Layer Steps In

A warehouse manager — logs the new item, tosses the old inventory list, sends it to storage.

SaveCardAsync in CardService

CODE
public async Task<PrayerCard> SaveCardAsync(
    PrayerCard card)
{
    await card.SaveAsync();
    _cache = null;
    return card;
}
PLAIN ENGLISH

This method takes a prayer card and saves it.

Tell the card to save itself to the database (Active Record pattern — the model knows how to persist itself).

Throw away the cache. The old list of cards is now stale because we just added or changed one. Setting it to null forces the next read to go back to the database for fresh data.

Return the saved card (it now has an Id if it was new).

File: PrayerApp/Services/CardService.cs — lines 57–62

💡
The cache invalidation pattern

This two-line pattern — await save; _cache = null; — appears in every write method of every service. It's the simplest form of cache invalidation: "something changed, throw away what I remembered."

04

The Model Saves Itself

A self-addressed envelope — the model knows its own mailing address and drops itself in the mailbox.

PrayerCard.SaveAsync()

CODE
public async Task SaveAsync()
{
    if (Id == 0)
    {
        await _dbService.InsertAsync(this);
    }
    else
    {
        UpdatedAt = DateTime.Now;
        await _dbService.UpdateAsync(this);
    }
}
PLAIN ENGLISH

The card saves itself. No external code needs to know how.

If the Id is 0, this card has never been saved — it's brand new. Insert it. The database assigns an Id automatically (AutoIncrement).

Otherwise, this card already exists — update it. Stamp the current time first so we know when it was last changed.

The _dbService is the shared static reference set at startup (Course 0, Module 8). It talks to SQLite.

File: PrayerApp/Models/PrayerCard.cs — lines 75–88

What happens in SQLite

The SQLite database receives an INSERT command and writes a new row to the PrayerCard table. The card now has a permanent home on disk.

1
InsertAsync runs the SQL

sqlite-net-pcl generates: INSERT INTO PrayerCard (Title, IsSystem, BoxId, ...) VALUES (?, ?, ?, ...)

2
AutoIncrement assigns an Id

SQLite picks the next available number (e.g., 42) and sets it as the card's Id.

3
The Id flows back to the object

sqlite-net-pcl reads the generated Id and sets it on the PrayerCard object. Now card.Id is 42.

05

The Data Comes Back

A boomerang — the data goes out to storage and comes back through the same layers, triggering updates on each return pass.

The user navigates back

After saving, the user goes back to the cards list. The cards page detects the return and calls RefreshAsync() on its ViewModel:

CODE
public async Task RefreshAsync()
{
    _cardService.InvalidateCache();
    var cards = await _cardService
        .GetCardsAsync();
    // ... update the list ...
}
PLAIN ENGLISH

The ViewModel's refresh method runs when returning to this page.

First, throw away the service's cached card list (because we just saved a new card).

Then ask the service for all cards. Since the cache was just cleared, this hits the database — and the database now includes our new card.

File: PrayerApp/ViewModels/PrayerCardsViewModel.cs — lines 554–560

GetCardsAsync rebuilds the cache

CODE
public async Task<IReadOnlyList<PrayerCard>>
    GetCardsAsync()
{
    if (_cache is not null)
        return _cache;

    var list = await PrayerCard.LoadAllAsync();
    _cache = new ReadOnlyCollection<PrayerCard>(
        list);
    return _cache;
}
PLAIN ENGLISH

If the cache still exists, return it immediately — but we just cleared it.

So this time: go to the database and load ALL prayer cards. The new card is in there now.

Wrap the fresh list in a read-only wrapper and store it as the new cache for next time.

File: PrayerApp/Services/CardService.cs — lines 16–26

The Screen Updates

A sports ticker tape — when the score changes, the display automatically scrolls to show the new result.

The ViewModel rebuilds its sections

After getting the fresh card list, the ViewModel creates BoxSectionViewModels — one per collection/box — and replaces the old list. XAML's ObservableCollection notifies the UI that data changed:

CODE
public ObservableCollection<BoxSectionViewModel>
    BoxSections
{
    get => _boxSections;
    private set => SetProperty(
        ref _boxSections, value);
}
PLAIN ENGLISH

BoxSections is the list of card groups shown in the CollectionView.

When a new list is assigned, SetProperty fires a PropertyChanged event. The XAML binding hears it and refreshes the list on screen.

The full round trip

1
View: User taps "Add Card"

XAML fires the bound NewCommand

2
ViewModel: Navigates to card form

User fills in title, taps Save

3
Service: Saves card, clears cache

CardService.SaveCardAsync runs card.SaveAsync(), sets _cache = null

4
Model: Persists to SQLite

PrayerCard.SaveAsync() calls _dbService.InsertAsync(this)

5
Return trip begins

User navigates back, RefreshAsync fires

6
Service: Rebuilds cache from DB

GetCardsAsync hits the database (cache was cleared), gets fresh list including new card

7
View: New card appears

BoxSections updated, PropertyChanged fires, CollectionView refreshes, card visible

💡
The golden rule of data flow

Data always flows down through the layers (View → ViewModel → Service → Model → DB) and back up the same path. No layer ever skips another. This predictability is what makes bugs findable — if data isn't showing up, check each layer in order.
