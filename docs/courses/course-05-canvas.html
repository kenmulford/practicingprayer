The Canvas

The visual layer — XAML layout, styling, theming, and making the app accessible to everyone.

This course covers everything the user sees — how pages are built with XAML, how colors and styles create the warm journal aesthetic, and how accessibility features ensure everyone can use the app.

01

XAML Fundamentals

An architect's blueprint — specifies where every wall, window, and light switch goes.

What is XAML?

XAML is the language that defines what each screen looks like. Instead of writing C# code to create buttons, you declare them in a markup file:

CODE
<ContentPage
    xmlns="...maui"
    x:DataType="viewModels:PrayerCardsViewModel">

    <Grid RowDefinitions="Auto,Auto,Auto,*">
        <SearchBar
            Text="{Binding SearchText}"
            Placeholder="Search cards..." />
    </Grid>
</ContentPage>
PLAIN ENGLISH

This is a full-screen page.

x:DataType tells the compiler which ViewModel this page binds to — enabling compile-time binding checks.

A Grid with 4 rows: three auto-sized rows and one that fills remaining space.

A search bar in the first row, bound to the ViewModel's SearchText property.

XAML + code-behind

Every .xaml file has a code-behind file (.xaml.cs). The XAML declares the layout; the code-behind handles lifecycle events like OnAppearing (page became visible) and OnDisappearing (page left).

02

Colors.xaml — The Palette

A chameleon's wardrobe — two complete outfits (light and dark), with descriptive labels for each piece.

Semantic color tokens

Instead of scattering hex codes like #6B7D5A everywhere, the app defines named color tokens:

CODE
<Color x:Key="Primary">#6B7D5A</Color>
<Color x:Key="PageLight">#FAF8F3</Color>
<Color x:Key="PageDark">#0d0e0c</Color>
<Color x:Key="CardLight">#FFFFFF</Color>
<Color x:Key="CardDark">#181a16</Color>
PLAIN ENGLISH

Primary — the app's signature warm green. Used for buttons, accents, and the app icon.

PageLight — the warm cream background in light mode.

PageDark — the deep charcoal background in dark mode.

CardLight/CardDark — card surface colors for light and dark themes.

⚠️
Never hardcode hex in XAML

If you see TextColor="#6B7D5A" in XAML, that's a bug. Always use TextColor="{StaticResource Primary}". Hardcoded hex means the color won't update if the palette changes, and it won't adapt to dark mode.

03

Styles.xaml — Reusable Recipes

A fashion designer's pattern book — sewing patterns for consistent garments.

Named styles

A named style lets you define a visual recipe once and reuse it across the app:

📦
PrayerCardBorder

The border style applied to every card — rounded corners, shadow, consistent padding.

🏷
FormLabel

The label style for form fields — specific font size, weight, and spacing above input fields.

🔘
Primary / Secondary button

Button styles with the right colors, padding, and corner radius for the app's aesthetic.

💡
The form convention

In this app, forms always use label above field layout with full-width inputs. Never side-by-side label+field. When telling AI to build a form, say "use the FormLabel style and vertical stack layout, label above field."

04

AppThemeBinding — Light & Dark

A house with two sets of curtains — one sheer for day, one blackout for night, the system picks which to draw.

How it works

AppThemeBinding lets you specify different values for each theme, right in the XAML:

CODE
<SearchBar
    BackgroundColor="{AppThemeBinding
        Light={StaticResource Secondary},
        Dark=Transparent}" />
PLAIN ENGLISH

The search bar background changes automatically:

Light mode: use the Secondary color (a soft green-gray).

Dark mode: use Transparent (blends into the dark background).

No if/else code needed — the framework handles the switch.

File: PrayerApp/Views/PrayerCard/PrayerCardsPage.xaml — line 31

⚠️
Known gotcha: AppThemeBinding + BindableLayout

AppThemeBinding inside a keyed Style doesn't reliably work for all BindableLayout children. Use inline color bindings for dividers inside BindableLayout DataTemplates. This is a known framework quirk.

CollectionView & Grouping

A bookshelf with labeled dividers — sections separate groups, EmptyView fills empty shelves.

Displaying lists of data

CollectionView is how the app shows the list of prayer cards. Each card is rendered from a DataTemplate:

1
ItemsSource

Binds to BoxSections — the grouped list of cards from the ViewModel.

2
GroupHeaderTemplate

A header for each section (e.g., the collection name: "My Prayers", "Work").

3
ItemTemplate

How each card looks — a Border with title, prayer count, and action chips.

4
EmptyView

What shows when there are no cards — a friendly "Add your first card" message.

06

Forms & Modals

A pop-up shop inside a mall — temporary overlay, must close before returning to the main floor.

Modal pages

When the app needs focused input (like the Quick Add prayer form), it opens a modal:

CODE
await Shell.Current.Navigation
    .PushModalAsync(page);
PLAIN ENGLISH

Open the page as a full-screen overlay. The user must close it to get back to the previous screen. Good for forms that require focused attention.

🔍
MAUI gotcha

The older Shell.PushModalAsync method was removed. Always use Shell.Current.Navigation.PushModalAsync instead. AI tools sometimes generate the removed version.

The tag picker modal — real-world example

The tag picker is a modal that opens when you tap the + button in prayer detail edit mode. It uses FlexLayout for flowing tag chips and rebuilds the layout in code-behind when tags change:

+
User taps + button

OpenTagPickerCommand fires, ViewModel creates a TagPickerViewModel and raises an event.

🔍
Search and select tags

Type to search, tap suggestions, use commas for batch entry. New tags created on-the-fly.

✅
Tap Done, chips update

Modal dismisses, parent syncs tag changes, code-behind rebuilds the FlexLayout with updated chips.

💡
Code-driven layout: when XAML isn't enough

The edit-mode tag chips are rebuilt in code-behind (not pure XAML binding) because the FlexLayout needs a + button as the last child after all tag chips. When the tag collection changes, RebuildEditTagChips() clears the FlexLayout, re-adds each chip Border, and appends the + button. This is an intentional exception to the "everything in XAML" rule.

Accessibility

Audio descriptions at a museum — narration for every exhibit, invisible labels for tour guides.

Screen reader support

SemanticProperties make the app usable by blind and low-vision users:

CODE
<ToolbarItem
    Text="Collections"
    AutomationId="Cards_Btn_Collections"
    SemanticProperties.Hint="Manage collections" />
PLAIN ENGLISH

A toolbar button that says "Collections."

AutomationId serves double duty: screen readers and UI test locators both use it.

Hint tells VoiceOver/TalkBack what tapping this button does — "Manage collections."

Programmatic announcements

CODE
public void Announce(string message)
{
    if (MainThread.IsMainThread)
        SemanticScreenReader.Announce(message);
    else
        MainThread.BeginInvokeOnMainThread(
            () => SemanticScreenReader
                .Announce(message));
}
PLAIN ENGLISH

Speak a message through the screen reader (e.g., "Loading" or "Content loaded").

If we're already on the main thread, announce directly.

Otherwise, switch to the main thread first (screen reader calls must happen on the UI thread).
